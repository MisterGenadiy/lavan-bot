fix(antiraid): не падать, если виновник нюка не в кэше участников

Анти-краш вызывал actor.ban()/actor.edit(roles=...), считая actor
всегда discord.Member. entry.user из guild.audit_logs() на деле
Optional[Union[Member, User]] — если виновник уже вышел с сервера
сразу после массового удаления каналов/ролей (типичный сценарий при
реальном нюке), discord.py возвращает обычный User, у которого нет
ни .ban(), ни .edit(roles=...). AttributeError не перекрывался
существующим except discord.Forbidden и тихо ронял listener —
анти-краш не срабатывал именно в том случае, для которого писался.

- guild.ban(actor, ...) вместо actor.ban() — принимает любой
  discord.abc.Snowflake, работает для Member и User одинаково.
- Снятие ролей теперь пытается сначала получить Member из кэша
  (guild.get_member), если actor пришёл как plain User, и только
  потом решает, возможно ли это действие в принципе.
- Уточнены type hints _get_actor_from_audit/_register_deletion:
  Member | User | None вместо Member | None.

perf(antiraid): троттлинг при массовом kick/ban на антирейде

Цикл по recent_members на срабатывание антирейда (action=kick/ban)
слал запросы к Discord API без пауз. На крупном рейде (десятки-сотни
вступлений разом) это упирается в rate limit, и часть
кик/бан-действий просто не применяется. Добавлена пауза 1 сек
каждые 5 действий.

fix(antispam,antiraid): периодическая очистка счётчиков активности

message_log, recent_actions, recent_mention_actions (antispam.py) и
join_log, delete_log (antiraid.py) — словари, ключи которых
(guild_id[, user_id]) добавлялись при каждом сообщении/вступлении/
удалении и никогда не удалялись. На долгоживущем процессе с большим
числом серверов/пользователей это неограниченный рост потребления
памяти.

Добавлена tasks.loop(minutes=15) в обоих cog'ах, которая убирает
записи неактивных ключей старше часа. Задача стартует в __init__,
корректно останавливается в cog_unload, дожидается
bot.wait_until_ready() перед первым запуском.

chore: удалить мёртвый utils/backup_core.py

utils/backup_core.py (206 строк, старая реализация save/restore)
никогда не исполнялся: Python при `from utils import backup_core`
всегда резолвит одноимённый пакет utils/backup_core/ раньше файла в
той же директории — подтверждено эмпирически импортом в интерпретаторе.
Весь публичный API файла (save_backup, restore_backup, has_backup,
get_backup_info, notify_guild_or_dm, _with_retry) уже продублирован
(и расширен: чекпоинты, дифф, ретеншн, безопасный restore) в пакете
utils/backup_core/. Удаление не меняет поведение бота, убирает
мёртвый код и коллизию имён, вводившую в заблуждение при чтении/грепе
проекта.
