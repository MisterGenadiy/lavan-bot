feat(backup): профессиональная система бэкапа и восстановления сервера

Переписан utils/backup_core.py (был один файл) в пакет utils/backup_core/
с разделением по ответственности:
  - models.py   — типизированная схема бэкапа (датаклассы) + RestorePlan/RestoreScope
  - storage.py  — файлы на диске (backups/{guild_id}/{backup_id}.json), уникальные
                  ID бэкапов, миграция старого "плоского" формата без metadata
  - capture.py  — снимок текущей структуры сервера (роли, категории, текстовые/
                  голосовые/форум-каналы, overwrites, эмодзи, стикеры, настройки
                  сервера) — логика /save и L.backup save
  - diff.py     — сравнение бэкапа с текущим состоянием сервера, сопоставление
                  сущностей по имени (а не по Discord ID), построение плана:
                  что создать / обновить / удалить / конфликты
  - restore.py  — применение плана к серверу + restore_with_safety() (авто
                  emergency-бэкап перед восстановлением, подготовка к rollback)
  - retry.py, notify.py — вынесены без изменений из старого backup_core.py

Что изменилось в поведении:
  - /save и L.backup save теперь сохраняют форум-каналы, эмодзи и стикеры
    (с картинкой в base64, чтобы восстановление работало даже после удаления
    оригинала), настройки сервера и метаданные (ID бэкапа, время создания,
    счётчики по типам сущностей). Старые ключи ответа (roles/channels)
    сохранены для обратной совместимости.
  - /load и L.backup restore больше не удаляют всё подряд: перед применением
    показывается план восстановления (embed: создать/обновить/удалить/
    конфликты), по умолчанию ничего не удаляется (safe-режим), удаление
    лишнего — отдельная явная опция (удалять_лишнее в /load, strict в
    L.backup restore).
  - Добавлена частичная область восстановления: только роли / только каналы /
    только категории / только права доступа (RestoreScope).
  - Перед каждым restore автоматически создаётся emergency-бэкап текущего
    состояния сервера — его ID показывается пользователю для возможного
    отката тем же механизмом восстановления.
  - Старые бэкапы в "плоском" формате (без metadata) продолжают читаться
    без какой-либо миграции файлов на диске — преобразуются на лету.

utils/embeds.py: добавлен build_restore_plan_embed() — общий рендер плана
восстановления для префиксных и слэш-команд, по аналогии с уже существующим
build_settings_embed().

.gitignore: добавлен паттерн backups/**/*.json под новую раскладку файлов
(backups/{guild_id}/{backup_id}.json), старый backups/*.json оставлен.

Не затронуто (намеренно): cogs/antispam.py, cogs/antiraid.py, cogs/antilink.py,
cogs/moderation.py, cogs/settings.py, cogs/verification.py, cogs/help.py,
utils/database.py, utils/moderation_utils.py, bot.py — изменения ограничены
функциональностью бэкапа/восстановления.

BREAKING: нет. Публичные функции backup_core (save_backup, has_backup,
get_backup_info, restore_backup, notify_guild_or_dm) сохранены с прежними
именами и совместимы по использованию из внешнего кода.
