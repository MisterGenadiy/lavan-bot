# Lavan-clone — бот защиты Discord-сервера

Аналог функционала бота Lavan: анти-спам, анти-рейд, анти-краш (защита от нюка),
система варнов с авто-наказаниями, модерация, профессиональная система
бэкапа/восстановления сервера, аудит-лог вотчер.

## Возможности

- **Анти-спам** — отслеживает частоту сообщений каждого пользователя, при превышении
  лимита применяет настроенное наказание (warn / mute / kick / ban).
- **Анти-рейд** — отслеживает всплески вступлений новых участников. При обнаружении
  атаки может закрыть сервер (lockdown), кикнуть или забанить недавно вступивших.
- **Анти-краш** — следит за журналом аудита на массовое удаление каналов/ролей и
  создание вебхуков (типичные признаки "нюка" сервера), банит виновника.
- **Аудит-лог вотчер** — дублирует ВЕСЬ аудит-лог сервера (создание/удаление/
  изменение каналов, ролей, прав, баны, настройки сервера, вебхуки, AutoMod
  и т.д., 67 типов событий) в выбранный канал — независимо от лимитов
  анти-краша, чисто для видимости происходящего.
- **Система варнов** — `L.warn`, авто-наказания по количеству варнов
  (`L.add-warn-action`), как `L.add-warn-action` в оригинале.
- **Модерация** — kick, ban, mute (timeout), unmute, unban, clear/purge.
- **Профессиональный бэкап сервера** — роли, категории, текстовые/голосовые/
  форум-каналы, права доступа, эмодзи, стикеры, настройки сервера, AutoMod-
  правила, вебхуки. Восстановление через предпросмотр плана изменений,
  partial restore, emergency-бэкап перед каждым restore, поддержка
  возобновления после краша бота, клонирование структуры на другой сервер.
- **Настраиваемый префикс** и канал логирования.

## Установка

1. Установите Python 3.11+.
2. Установите зависимости:
   ```bash
   pip install -r requirements.txt
   ```
3. Создайте приложение и бота на https://discord.com/developers/applications,
   включите **Privileged Gateway Intents**: `SERVER MEMBERS INTENT` и
   `MESSAGE CONTENT INTENT`.
4. Скопируйте `.env.example` в `.env` и вставьте токен бота:
   ```bash
   cp .env.example .env
   ```
5. Запустите бота:
   ```bash
   python bot.py
   ```
6. Пригласите бота на сервер с правами: `Administrator` (проще всего) либо
   набором: Manage Roles, Manage Channels, Manage Emojis and Stickers,
   Manage Webhooks, Manage Guild (для AutoMod и настроек сервера), Kick
   Members, Ban Members, Moderate Members (timeout), View Audit Log.
   Перед `/load`/`/rollback`/`/clone-template` бот сам проверяет, каких
   именно прав не хватает для выбранной области восстановления, и
   предупреждает об этом до применения плана.

## Основные команды (префикс по умолчанию `L.`)

| Команда | Описание |
|---|---|
| `L.settings` | показать текущие настройки |
| `L.prefix <новый>` | сменить префикс |
| `L.setlogchannel #канал` | назначить канал логов |
| `L.antispam on 6 6 mute` | вкл. антиспам: 6 сообщ./6 сек → mute |
| `L.antiraid on 8 10 lockdown` | вкл. антирейд: 8 вступлений/10 сек → lockdown |
| `L.auditwatch on/off [#канал]` | дублировать весь аудит-лог сервера в канал |
| `L.add-warn-action 3 mute 600` | за 3 варна → мут на 600 сек |
| `L.warn @user причина` | выдать предупреждение |
| `L.kick / L.ban / L.mute / L.unmute` | модерация |
| `L.backup save` | сохранить структуру сервера |
| `L.backup restore [область] [safe\|strict]` | восстановить из бэкапа (план + подтверждение) |
| `L.backup rollback [id]` | откатиться к авто-бэкапу перед последним restore |
| `L.backup resume [id] [discard]` | продолжить восстановление (или отменить чекпоинт без применения остатка) |
| `L.backup diff <id1> <id2>` | сравнить два бэкапа между собой |
| `L.backup info / list / duplicates` | инфо о бэкапе / список бэкапов / поиск дублей имён |
| `L.unlock` | снять lockdown после ложного срабатывания |
| `L.help` | список всех команд |

## Слэш-команды (бэкап/восстановление)

| Команда | Описание |
|---|---|
| `/save` | сохранить структуру сервера (роли, каналы, права, эмодзи, стикеры, AutoMod, вебхуки, настройки) |
| `/load [область] [удалять_лишнее] [бэкап]` | восстановить из бэкапа — показывает план изменений, требует подтверждения |
| `/rollback [бэкап]` | откатиться к авто-бэкапу перед последним restore (или к указанному) |
| `/resume [чекпоинт] [отменить]` | продолжить восстановление, прерванное крашем/перезапуском бота (или отменить чекпоинт) |
| `/backups` | список всех сохранённых бэкапов сервера |
| `/backup-diff <бэкап1> <бэкап2>` | сравнить два бэкапа между собой — что изменилось |
| `/find-duplicates` | найти роли/каналы/категории с одинаковыми именами |
| `/clone-template <id_сервера_источника>` | **[только владелец бота]** применить бэкап с другого сервера как шаблон |
| `/auditwatch включить:True/False [канал]` | дублировать весь аудит-лог сервера в канал |

## Структура проекта

```
lavan-bot/
├── bot.py                 # точка входа
├── cogs/
│   ├── settings.py        # настройки, префикс, антиспам/антирейд конфиг
│   ├── moderation.py       # warn/kick/ban/mute/clear
│   ├── antispam.py         # детектор спама
│   ├── antiraid.py         # детектор рейдов и нюков (анти-краш)
│   ├── auditwatch.py        # дублирование всего аудит-лога сервера в канал
│   ├── backup.py           # бэкап/восстановление сервера (L.backup)
│   ├── slash.py             # слэш-команды (/save, /load, /rollback, /resume, ...)
│   └── help.py              # справка
├── utils/
│   ├── backup_core/         # сохранение/сравнение/восстановление структуры сервера
│   │   ├── models.py        # типизированная схема бэкапа (v3) + RestorePlan/RestoreScope
│   │   ├── capture.py       # снимок текущего сервера: роли, каналы, эмодзи, AutoMod, вебхуки
│   │   ├── diff.py          # сравнение бэкапа с сервером (план) + сравнение двух бэкапов + поиск дублей
│   │   ├── restore.py       # применение плана, emergency-бэкап, прогресс, чекпоинты
│   │   ├── storage.py       # файлы на диске (gzip), ID бэкапов, ретеншн, миграция старого формата
│   │   ├── checkpoint.py    # чекпоинты восстановления для /resume + автоочистка забытых
│   │   ├── locks.py         # per-guild asyncio.Lock против гонки /save и /load
│   │   ├── permissions.py   # pre-flight проверка прав бота перед restore
│   │   ├── retry.py         # повтор временных сбоев Discord API
│   │   └── notify.py        # уведомление пользователя устойчивое к удалению канала
│   ├── embeds.py             # embed-билдеры: план восстановления, лог действий с бэкапом
│   ├── database.py          # SQLite-обёртка для настроек/варнов
│   └── moderation_utils.py  # применение наказаний, проверка прав
├── tests/                    # юнит-тесты backup_core (pytest, без реального discord-соединения)
├── .github/workflows/ci.yml  # синтаксис + тесты на каждый push/PR
├── backups/                  # backups/{guild_id}/{backup_id}.json.gz + _checkpoints/ (не в git)
├── logs/                      # runtime-логи бота (не в git)
├── requirements.txt
├── requirements-dev.txt      # только для разработки (pytest)
├── .env.example
└── .gitignore
```

## Разработка и тесты

```bash
pip install -r requirements.txt -r requirements-dev.txt
pytest -q
```

Тесты покрывают `utils/backup_core` (сравнение бэкапа с сервером, сопоставление
по имени, применение плана, AutoMod/вебхуки, gzip-хранилище, чекпоинты/resume,
per-guild локи, проверку прав бота, формат embed-сообщений) через лёгкие
фейковые объекты (`tests/fakes.py`) — реальное подключение к Discord не
требуется. CI (`.github/workflows/ci.yml`) прогоняет их на Python 3.11/3.12
при каждом push/PR в `main`.

## Примечания

- Все настройки хранятся в SQLite (`data.sqlite3`), создаётся автоматически.
- Бот должен иметь роль **выше** ролей, которые он должен мутить/банить.
- Для работы анти-краша нужен включённый journal аудита (по умолчанию доступен
  всем серверам) и право `View Audit Log` у бота.
- **Аудит-лог вотчер** (`/auditwatch`, `L.auditwatch`) дублирует ВСЕ записи
  аудит-лога в канал — это отдельная функция от анти-краша: анти-краш
  реагирует на превышение лимита удалений и наказывает, вотчер просто
  показывает происходящее без вмешательства.
- `/load`, `/clone-template` и `L.backup restore` показывают план изменений
  (что создастся/обновится/удалится/какие конфликты), пред-проверку прав бота
  и требуют подтверждения; по умолчанию ничего не удаляется (`safe`-режим) —
  удаление лишнего нужно явно включить (`удалять_лишнее` в `/load` или режим
  `strict` в `L.backup restore`). Перед применением плана автоматически
  создаётся резервный (emergency) бэкап текущего состояния сервера.
- `/rollback` и `L.backup rollback` восстанавливают этот emergency-бэкап —
  удобно, если сам restore пошёл не так, как ожидалось.
- Если бот упал или был перезапущен посередине восстановления, `/resume` /
  `L.backup resume` продолжает с того места, где всё прервалось, не повторяя
  уже применённые пункты плана — прогресс пишется на диск по ходу restore
  (чекпоинт автоматически удаляется при успешном завершении). Если
  продолжать не нужно (проблему решили вручную), чекпоинт можно отменить:
  `/resume отменить:True` или `L.backup resume <id> discard` — уже
  применённые до сбоя пункты при этом не откатываются, отменяется только
  применение оставшихся. Чекпоинты, про которые забыли (старше недели),
  удаляются автоматически при следующем запуске бота.
- `/clone-template` (**только владелец бота**) применяет бэкап ОДНОГО сервера
  как шаблон к ДРУГОМУ — например, чтобы быстро развернуть стандартную
  структуру на новом сервере.
- `/backup-diff` и `L.backup diff` сравнивают два сохранённых бэкапа между
  собой — «что изменилось на сервере между датой A и датой B», без
  затрагивания текущего состояния сервера.
- `/find-duplicates` и `L.backup duplicates` находят роли/каналы с одинаковыми
  именами — restore сопоставляет сущности по имени, а не по Discord ID, и такие
  дубликаты не получится обновить автоматически (они попадут в «конфликты»).
- Бэкап включает роли, категории, текстовые/голосовые/форум-каналы, права
  доступа, эмодзи, стикеры, AutoMod-правила, вебхуки (без токена — Discord
  выдаёт новый при пересоздании) и доступные через API настройки сервера.
  **Известное ограничение:** треды и посты внутри форум-каналов не сохраняются —
  бэкапит саму структуру каналов/категорий, а не их содержимое или треды
  внутри них (они обычно эфемерны и не считаются частью "структуры" сервера).
- Бэкапы хранятся сжатыми (`gzip`), старые несжатые файлы по-прежнему
  читаются без какой-либо ручной миграции.
- Хранится не более 10 обычных и 5 emergency-бэкапов на сервер — старые
  удаляются автоматически при каждом новом `/save`/restore
  (`storage.MAX_REGULAR_BACKUPS` / `MAX_EMERGENCY_BACKUPS`).
- Одновременные `/save` и `/load` на одном сервере не пересекаются —
  используется блокировка на уровне сервера; разные серверы друг друга
  не блокируют.
- Каждое действие с бэкапом (`save`/`load`/`rollback`/`resume`/
  `clone-template`) дублируется в mod-log-канал — кто и когда что применил.
